### 参照整合性の崩壊
- 他のテーブルと紐付けをするキー（外部キー）が存在する場合紐付け先が存在しないといけない
- 紐付け先が存在する状態を参照整合性という

### 参照整合性が崩壊する4つのパターン
- 参照先のテーブルの行がDELETEされる
- 参照先のテーブルの行が別の値にアップデートされる
- 存在しない行を参照する行をINSERTする
- 存在しない行を参照する値にUPDATE

### 外部キー制約
- 外部キーを追加すると参照整合性の崩壊が起きた時にエラーが発生し参照整合性を維持できるようになる


## カラムの型、インデックス

### データタイプの設計ができていない場合
- DBのパフォーマンスが悪化する可能性がある
- 正確な値をDBに格納できないかもしれない

### `CHAR`, `VARCHCAR`
- 文字列を扱う際には`CHAR`(固定長文字列)`VARCHAR`(可変長文字列)がある。
- 作成する際には最大の文字列サイズを設定する

```sql
colmun_name_1 VARCHAR(255)
colmun_name_2 CHAR(10)
```

### `CHAR`
- DBにデータを格納する際にあらかじめ格納するデータの領域が固定で確保される
- 格納する値が指定した長さより大きくてもディスク領域が埋まる
- 格納できる最大文字列255文字

### `VARCHAR`
- DBに格納する際にデータに応じて確保されるデータ領域を調整する
- 格納できる最大文字列65535文字


文字列の長さが固定されている場合は`CHAR`を用いる


### `TEXT` 
- 基本的な使い方はVARCHAR(可変長文字列)と同じだが、VARCHARで格納できないサイズの文字列を格納する場合に用いる
- 実際に格納されたデータが65535バイトより少ない場合ディスク使用量は少なくなる
- インデックスを作成することができない


## インデックスについて
- インデックスとは...テーブルの特定のカラムに対する索引
- インデックスを用いることで特定のレコードへのアクセスを高速で行うことができるようになる

使用例
```sql
CREATE INDEX {index_name} ON {table_name}(column1, column2)
DROP INDEX {index_name}
SHOW INDEX FROM {table_name}
```

- 関数インデックス
  - 関数に対するインデックス。関数の結果を用いた処理を高速化する場合に用いる
```sql
SHOW INDEX FROM students
CREATE INDEX idx_students_lower_name ON students((LOWER(name))) #関数インデックスの作成
```

- 複数カラムに対するインデックス
  - 複数のカラムや関数に対してインデックスを作成することができる
```sql
CREATE INDEX index_name ON table_name(column1, (function_name(column1, column2, ...)
```

- ユニークインデックス
  - カラムの組み合わせに対してインデックス+ユニーク制約をつける
```sql
CREATE UNIQUE INDEX idx_users_uniq_first_name ON users(first_name)
INSERT INTO users(id, first_name) VALUES(2, "ABC");

INSERT INTO users(id, first_name) VALUES(3, "ABC"); #同じ値を入れることができない
Query 1 ERROR at Line 1: : Duplicate entry 'ABC' for key 'users.idx_users_uniq_first_name'
```


## テーブル設計 

### テーブル設計の目的
- データの冗長性を減らして、使用するディスクスペースを減らす
- テーブル内のデータの正確さを維持する
- 適切にテーブルを分割して、効率的にアクセスできるようにする

### データ中心アプローチとは(DOA)
- システム設計の考え方の一つ
- まずどういうデータを利用するのかを洗い出す
- 次にどのようにデータを扱うのかを考えて処理を書き出すアプローチ(圧倒的に主流、POAは非効率)
- 他方で処理を洗い出してその後にデータを設計するアプローチがプロセス中心アプローチという(POA)

### 概念スキーマを作成する上で行うこと
- 概念スキーマ設計では必要なデータと格納するテーブルを決める
- テーブルを分割して外部キーを作成しどのテーブルとどのテーブルを紐づけるか決める

### 第３正規系での問題
- 第３正規系まで進めるとテーブルを効率的に分割することができテーブル構造がわかりやすくなる
-　また総データ量を抑えることはできる
- ただしテーブルの結合が増えるため処理に時間がかかる

### テーブル定義のコツ
- 主キーは必ずつける。主キーには数値型(INT)または固定長文字列を用いる(CHAR)
- レコードが挿入された時刻(created_at)と更新された時刻(updated_at)は必ずつける
- 文字列数型や数値型は大きすぎる方を設計しない
  - 文字列型や数値型に大きずぎるものを設計するとデータ使用量が多くなるため必要十分なサイズに設計する
- カラムの制約、デフォルト値、コメントはきちんとする

## Viewの作成
- Viewとはテーブルとテーブルを紐付けたり、特定のカラムだけを取り出したりして作成された仮想的なテーブルのこと

```sql
CREATE VIEW stores_name_view AS
SELECT 
	st.`name` AS store_name,
	it.name as item_name 
FROM
	stores as st
INNER JOIN
	items AS it
ON
	it.store_id = st.id

SELECT * FROM stores_name_view
```


## DBとパフォーマンスについて
- システム開発においてパフォーマンスを上げることは大事だがパフォーマンス向上を握るのがDBである
- 運用を続けていくとDBのデータ量が増加する。SQLが悪いとDBのデータ量に比例して処理時間が伸びる
- アプリケーションサーバーはスケールアウトが容易だがDBはスケールアウトが難しい

### SQL実行の流れ
- SQLはそのまま実行されるのではなく最も効率的に処理される内容(バイナリのプログラム)を最終的に作成して実行する

### ソフトパースとハードパース
- パースの際に共有プールをチェックして以前に実行された計画がないか確認する。
- もし実行計画が以前に作成されている場合は最適化、実行計画の作成を飛ばしてそのまま実行される、
- そのことをソフトパースという

- SQLを初めて実行する場合はハードパースとなり時間がかかる

->バインド変数を使って意図しないハードパースを避ける

バインド変数例
```sql
SET @customer_id=7;
SELECT * FROM customers WHERE id =@customer_id
```

## コストベースオプティマイザとは
- DBのオプティマイザがSQLから最も効率的な処理の実行方法(実行計画)を作成する
- 効率的な処理方法を作成するには最新の統計情報が必要となる
- コストベースオプティマイザとはインデックスやデータ量などの情報をもとに複数に実行計画のコストを計算し最も低いコストで実行される


### 統計情報を集める
- テーブルの統計情報は自動的に収集されるがコマンドを用いて手動で収集することもできる
- 急なテーブルのデータ量の増加があったときは手動で統計情報を更新すればいい

```bash
# 統計情報を収集するコマンド
ANALYZE TABLE テーブル名

# テーブルの統計情報一覧を確認
SELECT * FROM mysql.innodb_table_stats;

# インデックス統計情報の一覧を確認するコマンド
SELECT * FROM mysql.innodb_index_stats
```
### 実行計画とは
SQLが結果を得るための処理の具体的な内容と実行順序を示したもの
`EXPLAIN SQL文`SQLを実行せず実行計画だけを表示する
`EXPLAIN ANALIZE SQL文` SQLを実行して実行時間も含めた計画実行を表示する


```sql
EXPLAIN ANALYZE SELECT * FROM customers;

-> Table scan on customers  (cost=52522 rows=495432) (actual time=1.08..184 rows=500000 loops=1)
```

## 代表的なテーブルのアクセス方法について

### テーブルフルスキャン

テーブルから各行全てを確認してデータを取り出す
大量(約110万以上)のデータがテーブルに存在する場合、非常に時間がかかる

テーブルがフルスキャンされる例
```
SELECT * FROM users
SELECT * FROM users WHERE name=xx
SELECT * FROM users WHERE name LIKE "%xx%"
```

### インデックスユニークスキャン
- テーブル内のユニークな一件をインデックスを用いて検索する
- 一件が見つかれば検索終了のため高速に処理が終わる

主キーユニークキーで検索をかける
```sql
SELECT * FROM users WHERE id=10
```

### インデックスレンジスキャン
- インデックスを用いて複数の行を取得する際の検索方法
- 対象となるレコード件数に応じて処理時間が変わる

ユニークでないキーで絞り込む
```
SELECT * FROM users where age = 50;
```

## どちらを使うべきか

1 **データ量が少ない場合（約1万件以下）**の場合、インデックスを使っても使わんくても処理時間は変わらない
2 **絞り込めるデータの量の割合に応じてフルスキャンとインデックススキャンのどちらが有効か決まる**
- 20%以上...フルスキャンを用いいる
- 1-20%...ケースバイケース...(一説では15%以下はインデックススキャンの方が高速)
- 1%未満...インデックススキャンを使う
